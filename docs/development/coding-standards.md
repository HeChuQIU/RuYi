# 开发规范

## 代码规范

### 注释规范

- **注释范围**：仅对类（Class）和函数（Function/Method）编写注释，禁止为单行代码或代码段添加注释
- **注释原则**：如果代码逻辑需要注释才能理解，应通过函数拆分和重构提升代码可读性，而非添加注释
- **注释内容**：注释应说明"为什么"（Why），而非"是什么"（What），代码本身应清晰表达"是什么"

### 函数设计原则

- **控制流函数**：业务处理函数应作为控制流函数，仅包含函数调用和控制流语句（条件、循环等）
- **函数拆分**：业务逻辑的具体实现应通过函数拆分，逐层分解为单一职责的小函数
- **单一职责**：每个函数应只做一件事，函数长度建议不超过50行

### 命名规范

- **自解释性**：函数和变量命名应具有自解释性，通过命名即可理解其用途和含义
- **命名风格**：使用有意义的名称，避免缩写和单字母变量（循环变量除外）
- **布尔命名**：布尔变量和方法应使用 `is`、`has`、`can` 等前缀，例如：`isValid`、`hasPermission`

### 前端代码规范

- 遵循 Vue 3 官方风格指南
- 使用 ESLint + Prettier 进行代码格式化
- 组件命名使用 PascalCase
- 文件命名使用 kebab-case

### 后端代码规范

- 遵循 C# 编码规范
- 使用 EditorConfig 统一代码风格
- 类名使用 PascalCase
- 方法名使用 PascalCase
- 私有字段使用 _camelCase

## Git提交规范

使用约定式提交（Conventional Commits）：

- `feat:` 新功能
- `fix:` 修复bug
- `docs:` 文档更新
- `style:` 代码格式调整
- `refactor:` 代码重构
- `test:` 测试相关
- `chore:` 构建/工具链相关

### 提交示例

```bash
feat: 添加用户登录功能
fix: 修复项目列表加载问题
docs: 更新API文档
style: 格式化代码
refactor: 重构用户服务
test: 添加单元测试
chore: 更新依赖版本
```

## 分支管理

- `main`：主分支，稳定版本
- `develop`：开发分支
- `feature/*`：功能分支
- `fix/*`：修复分支
- `hotfix/*`：紧急修复分支

### 分支命名规范

- 功能分支：`feature/功能名称`，例如：`feature/user-authentication`
- 修复分支：`fix/问题描述`，例如：`fix/login-error`
- 紧急修复：`hotfix/问题描述`，例如：`hotfix/security-patch`

- 确保代码通过所有测试
